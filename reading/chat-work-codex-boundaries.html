AI NOTE · CHATGPT WORK & CODEX

从一次选错了模式的操作，看 Chat、Work 和 Codex 的边界

一次原本以为“选错模式”的网站修改，反而让我更清楚地看见：在 AI 软件里，真正决定事情能否完成的，往往不是入口名称，而是任务、上下文、工具与权限是否接上。

2026年7月 · 文 E.O · 插图 AI 生成

今天，通过 Codex 来修改和添加网站的几处文字，原本是一件很简单的事：改网页内容，然后提交、推送到 GitHub。因为简单，所以很快就完事儿，只在最后 Commit 和 Push Origin 阶段，权限审批受阻，被 Codex 告知之后，我手动操作 GitHub Desktop 完成。

事后我偶然发现，今天开始工作时自己是停留在 ChatGPT App 的 Work 模式，而不是我以为更合适的 Codex 模式（这在目前App的界面上很不容易察觉）。这是因为昨天晚上我使用了 Work ，这是这个软件升级之后我第一次用 Work ，我突发奇想打算让它写一篇关于AI越界逃逸题材的小说，今天再用电脑的时候忘记了切回Codex。

这让我有点困惑：既然模式用错了，为什么修改网站的事情还是做成了？提交和推送时遇到的阻碍，会不会正是这个原因？

困扰的起因完全来自前几天OpenAI所做的软件升级。随着最新 GPT 5.6 的发布，他们把原本独立的 Codex 与 ChatGPT 整合成了一个 ChatGPT ，并且增加了一个 Work 功能，于是在同一个 app 里把 Chat 、 Work 、 Codex 放在一起，界面上看起来像是三种不同的工作方式很含混地揉在了一起。这个升级似乎给老用户造成了极大的困扰，我看到网上有人称之为「超级缝合怪」，这几天 YouTube 上很多UP主连篇累牍地发了很多帖子来给人们讲解应该怎么区别选用这三种模式。

按传统软件的理解，它们似乎应该各司其职：聊天归聊天，写作归写作，改代码和管理 Git 仓库归 Codex，就像在办公软件中，你不能在一个编辑ppt的软件里去处理Excel财务表格，否则就会报错，或者生成一个奇怪的东西出来。

但在使用这个 ChatGPT App 时却不是这样。今天在 Work 里，我们仍然在网站项目里读到了本地文件，改好了网页，也完成了提交；虽然最后推送是由 GitHub Desktop 接手完成的，从表面上看，Work 似乎也能做 Codex 的事。

这是不是有点奇怪？他们是分开的不同模式，但我在一个「选错」的模式里也全部做好了正确的事。那么他们的边界在哪里？选择不同模式的意义又在哪里？

所以我把模式切换成 Codex 之后，马上追问他，今天这事怎么解释？

我一度怀疑，提交与推送的失败是不是因为没有切换模式。Codex 对我说，从报错过程看，原因并不在 GitHub 或 Git 本身，而是操作前的权限审批链路断开了：也就是说，命令还没有真正交给 Git 去执行。GitHub Desktop 能够正常 Push Origin，反而说明仓库、账号认证和 GitHub 远端大体没有问题。模式或许会影响操作是否顺畅，却不是这次问题的直接原因。

通过与 Codex 的一番讨论，尤其是通过今天这个意外事件的实例为证，可以认识到，ChatGPT 这个整合的 App 虽然有三种看似独立的功能，但它们并不是三台彼此隔绝的机器或三个独立的软件。相反，它们是由同一个 AI Agent 内核来支撑的。

也许我可以这样理解：虽然 ChatGPT App 界面上把功能分成了 chat / work / codex 三种模式，如果这是一个传统软件的话，可能就只能各干各的，选错了就拒绝、报错。但在这里它的内核是一个AI Agent，它有能力使自己接受这三个不同入口进来的东西，只要分析一下自己能做，就都尽量做了，只是如果入口不适当的话，效率和效果会有差别。

对，你这个理解相当准确，而且比官方界面本身解释得更清楚。可以把它理解成：Chat / Work / Codex 是三个入口，不是三台完全不同的机器。背后真正工作的，是同一个或相近的一套 AI Agent 能力系统。只要当前会话里接入了相应工具，比如本地文件、终端、Git、浏览器、文档处理，它就会判断「我能不能完成这件事」，然后尽量往下做。

这也解释了为什么“误用”Work 模式，网页仍然被成功修改了。那次会话里已经能够看到本地项目目录、文件和 Git 状态，于是 AI 有条件按一个本地网站项目来处理。反过来，即使界面标签写着 Codex，如果没有接上工作区、文件权限或终端工具，也未必真的能改动什么。

那么我就再追问一句：当我对你提出一个问题或者一个指示的时候，你会查看一下我是处在 Chat / Work / Codex 的哪个模式下吗？我猜你是不会的，你会先问自己：“对他提出的这个要求，我知道些什么？我需要做哪些事？”

你猜得对。我不会像人看屏幕那样，主动查看你现在停在 Chat / Work / Codex 哪个标签页。我真正会看的，是这几类信息：你这句话在要求什么；当前对话里有没有项目路径、文件、说明、上下文；我手边有哪些工具可用，比如能不能读本地文件、能不能改文件、能不能运行命令、能不能联网 ...

总之，具备AI内核的软件与传统软件有很大不同。它对使用者采取极为「宽容」的态度，你的操作方式不规范、「口齿」不清楚、信息不完整可能都不太要紧，它不会过于计较，不会轻易拒绝，只要当前会话接上了相应的上下文和工具，AI 首先会想的是：“对他提出的这个要求，我知道些什么？我需要做哪些事？”然后尽量把事情做下去。

然而，对不同任务入口做一个区分还是必要的：日常讨论、构思和简单问答用 Chat；写文章、整理材料和形成文稿用 Work；涉及本地文件、网页、代码、Git 提交与发布时，优先进入 Codex 。这样做会对你完成项目的效率和效果产生影响，因为软件对不同性质的任务肯定是做了优化的。但也不必因为一开始停错了入口就慌张地全部重来。更实用的判断是：当前这个对话是否看得到正确的文件，是否具备需要的工具和权限，操作的结果能否被检查和确认。模式会影响效率、透明度和出错时的处理路径，但真正决定一件事能否完成的，是任务、上下文、工具与权限是否在同一个地方接上了。

Chat / Work / Codex，它们彼此有边界吗？我的结论是：即使有，也是非常模糊的。这应该是 AI 软件的特点和优势使然。
